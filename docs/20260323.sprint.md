# March Sprint

## Goal
- Reduce roadmap risk before adding more features.
- Make the sharing and recovery flows more reliable.
- Expand test coverage around the flows most likely to regress.

## Current test status
- `npm test` passes on March 23, 2026.
- Coverage is still light around hash recovery, finalized-game reload, submission import edge cases, and larger share payloads.

## Priority order
1. Fix broken or misleading game recovery from bad hashes.
2. Remove name-matching from submission import.
3. Fix the finalized-game share code race on reload.
4. Decide the future of the URL-hash architecture for scale.
5. Add test coverage for all of the above.
6. Revisit mobile ranking interaction and player dropout handling.

## Bugs to squash

### 1. Bad hashes silently fall back to local data
- Problem: if a shared game hash is malformed or stale, the app quietly loads whatever is in `localStorage`.
- Risk: users can open the wrong game and not realize it.
- Implementation direction:
- Add a visible error state when hash decoding fails.
- Do not silently hydrate a player/reveal session from unrelated local data.
- Decide whether setup view may still fall back to local draft data, but keep player/reveal links strict.
- Clear or ignore the invalid hash after showing the error so the app does not keep retrying it.
- Tests to add:
- Invalid player hash shows an error.
- Invalid reveal hash shows an error.
- Setup without a hash may still load a local draft if that is the intended behavior.

### 2. Finalized game code can render empty after reload
- Problem: `renderAll()` triggers async share-code generation but reads `state.shareEncoded` immediately.
- Risk: hosts can reload and see an empty backup code or inconsistent finalized state.
- Implementation direction:
- Make finalized-game initialization await share-code generation before rendering dependent fields.
- Keep share-link generation in one place so the state is not updated opportunistically from several paths.
- Tests to add:
- Reloading a finalized game restores the backup code.
- Reloading a finalized game keeps the share URL populated.

### 3. Submission import depends on exact player-name matching
- Problem: imported lines use `name:payload`, and the host resolves submissions by exact display name.
- Risk: casing changes, renames, spacing differences, or localization changes break import.
- Implementation direction:
- Switch the import/export format to identify players by `playerId`.
- Keep the display name only as a convenience label, not the primary key.
- Normalize names in setup validation so duplicates are checked case-insensitively.
- Consider backward compatibility for old `name:payload` lines if you want a soft migration.
- Tests to add:
- Import works when casing differs in the display name.
- Duplicate names with different casing are rejected in setup.
- Old payloads are either supported explicitly or rejected with a clear error.

### 4. Ranking interaction is weak on phones
- Problem: dragging does not work well on touch devices.
- Risk: the core player flow is harder on the devices people will actually use during a party.
- Implementation direction:
- Treat tap-based up/down controls as the primary mobile interaction.
- If drag-and-drop stays, make it progressive enhancement rather than the only good UX.
- Add a mobile-oriented UI test if feasible, otherwise cover ranking updates in unit tests.

## Features to think through

### 5. Sharing architecture is too heavy for scale
- Problem: both player onboarding and reveal links depend on compressed URL hashes.
- Risk: there is a hard ceiling on players, questions, and future metadata.
- Options:
- Short-term: keep hashes, but shrink the payload format for player submissions.
- Medium-term: move reveal state and submissions to a host-local store keyed by a short game code.
- Long-term: introduce a lightweight backend or realtime store if collaborative collection becomes a real product feature.
- Decision to make before implementation:
- Is this app staying fully offline/local-first, or is a small backend acceptable?

### 6. Player dropout after game creation
- Problem: current flows assume all finalized players will submit.
- Risk: one no-show can block the reveal.
- Implementation direction:
- Define whether the host can mark a player inactive after finalization.
- Recompute rankings, validation, and reveal data against the active player set.
- Decide how previously submitted rankings should be transformed if a player is removed.
